## 1. 🎉서비스 소개
꼬북코딩은 개발자들의 재능과 열정을 모아 지식과 기술을 나누며 새로운 프로젝트를 함께 만들어 나가는 공간입니다. 👋 홈 피드에서 게시글을 보고 follow한 사람들의 목록 확인과 댓글, 좋아요 기능을 통해 소통하고 나의 소식도 공유해보세요.

꼬북코딩은 개발자들을 위한 온라인 커뮤니티 플랫폼입니다. 개발자들간의 관계를 맺고, 자신의 지식을 공유하세요. 꼬북코딩과 함께 커리어를 성장시킬 수 있어요. 꼬북코딩을 통해 더 나은 미래를 만들 수 있는 개발자가 되길 응원합니다.🙌

커뮤니티를 통해 코드 오류 질문 & 답변부터 스터디원 모집, 강의나 기기등 상품을 사고팔고 재능기부를 통한 봉사활동도 가능합니다.🎶

## 9. 👿겪었던 문제와 해결과정
  <details>
    <summary><strong>이지은</strong></summary>

1. 댓글을 작성할 때마다 게시글 댓글 개수를 최신 값으로 바꿔주기 위해 post api를 조회해야 할까?
    
  - Post(게시글)를 공통 컴포넌트로 만들어서 사용하는데, 이 공통 컴포넌트에 댓글 개수도 포함되어 있었습니다. 게시글 상세화면에서 댓글을 작성하거나 삭제할 때 마다 댓글 개수도 그에 맞게 조정이 돼야 했는데, 이를 위해 post api를 계속하여 호출하면 비효율적이라는 생각이 들었습니다. 이 문제를 해결하기 위해 post 공통 컴포넌트에 댓글의 값을 props로 전달하여 배열로 값을 담고 있는 댓글의 length로 댓글 개수만 수정 되도록 처리하였습니다.
  </details>
  <details>
    <summary><strong>이재영</strong></summary>

1. 컴포넌트 분리 과정

  - 유독 내 페이지에서 겹치는 디자인이 많았다. 코드의 중복을 없애고 재사용 하기 위해 컴포넌트를 분리하는 과정에서 어려움을 겪었다. 얼마나 세세하게 해야될지 어디서부터 어떻게 분리해야 될지 생각을 많이 했던 거 같다. 구글에 검색을 하고 강사님들께 질문을 한 결과 여러 페이지에서 자주 쓰이는 컴포넌트들은 세세하게 분리해 common 폴더에 따로 저장하고 그렇지 않으면 최대한 덩어리로 분리하는 게 좋은 방법이라고 생각한다.

2. ContextAPI

1. search검색시 upDateData 함수를 계속 호출하는 문제

  - input창에 사용자 검색을 할때 upDateData함수를 계속 호출하는 문제가 있었습니다. 왜 그런지 확인해보니 의존성배열에 inputValue가 바뀔때마다 upDateData함수를 호출하도록 초기에 해놨기 때문에 api를 실시간으로 불러오고 있었습니다.
  - useEffect를 사용해 upDateData지연시키기

  ```jsx
  useEffect(() => {
    const delay = setTimeout(() => {
      if (inputValue) upDateData();
    }, 200);
    return () => clearTimeout(delay);
  }, [inputValue, upDateData]); //inputValue 키워드 변경시 api 호출함
  ```
  
  - useEffect에서 사용하고 있는 초기값이 변경될때에만 setTimeout이 불릴 수 있도록 의존성 배열에 inputValue와 upDateData를 넣어주었습니다.
  - setTimeout을 걸어준 이유는 예를 들어서 “ㅋ ㅗ ㄷ ㅣ ㅇ” 라는 글씨를 쓴다면 총 5회 upDateData함수 호출을 할것이고 이는 불필요한 데이터 호출을 야기합니다.
  - 때문에 딜레이를 두어서 입력이 다 끝나면 호출을 하기로 했습니다.
    
2. 상품 신규 등록과 상품 수정 분기처리하기
    
  - AddProduct페이지에서 상품 신규 등록과 수정을 하는것에서 분기처리가 필요했습니다.
  - 이 과정에서 무엇을 기준으로 처리 해야하는지 시간이 오래걸렸는데 결론은 useParams로 선택된 상품의 id값을 넘겨서 분기처리를 했습니다.
  
  ```jsx
  const itemID = useParams().id; //profile에서 선택된 상품의 id값
  
  //처음 상품 정보 가져올 때 id값으로 분기 처리
  useEffect(() => {
    if (itemID) productDetailInfoData();
  }, [itemID]);
  
  //8.2 선택된 상품 상세 가져오기 api
    const productDetailInfoData = useCallback(async () => {
      try {
        const res = await fetch(`${URL}/product/detail/${itemID}`, {
          method: "GET",
          // 요청 헤더, 바디 생략
        });
        if (!res.ok) throw new Error("네트워크 문제가 발생했어요.");
        const selectedItem = await res.json();
        // ...
      } catch (error) {
        console.error("🚫데이터를 불러오는데 에러가 발생했어요", error);
      }
    }, [itemID]);
  ```
  
  - useEffect는 처음 렌더링시 무조건 한번은 실행되며 이때 itemID값이 있을 경우 productDetailInfoData() 함수를 호출합니다. 또한 itemID값은 상시로 바뀌기 때문에 바뀔때마다 해당 함수를 호출합니다. (함수는 선택된 상품의 상세 api)
  
  ```jsx
  //product API 함수
  const handleAddProductSubmit = useCallback(
    async (e) => {
      if (e) e.preventDefault();
      try {
        //이미지를 넣어주기 위해서 body전에 삽입하고 imgSubmit을 호출
        const uploadResult = await imgSubmit();
  
        //📌상품id가 없을 경우 신규 상품 등록
        if (!itemID) {
          const res = await fetch(`${URL}/product`, {
            method: "POST",
            // 요청 헤더, 바디 생략
          });
          // ...
          if (!res.ok) throw new Error("네트워크 문제가 발생했어요.");
        }
        //📌상품id가 있을 경우 수정 api 적용
        else {
          const res = await fetch(`${URL}/product/${itemID}`, {
            method: "PUT",
            // 요청 헤더, 바디 생략
          });
          // ...
          if (!res.ok) throw new Error("네트워크 문제가 발생했어요.");
        }
      } catch (error) {
        //에러 핸들링 로직 생략
      }
    },
    [의존성 배열 생략]
  );
  ```
  
  - onSubmit속성에 할당된 handleAddProductSubmit 함수는 버튼 클릭시 실행되며 이때 itemID의 값을 구분해서 어떤 형태로 데이터를 전송할 것인지 구분하게 해주었습니다.
  - 이렇게 구조를 짜고보니 itemID분기처리하면 되는거 아니었나? 라고 생각이 드는데 처음 해볼때는 정말 막막했습니다. 하지만 구현하고 보니 이런 방식으로도 하는거구나하는 생각과 함께 기능 구현에 치중을 두어 반복적인 api의 코드가 보여서 다음엔 더 효율적으로 개선해보는 방법을 고려해보고 싶습니다.

1. 프로필 수정페이지 - API 메서드의 분리와 적용
  - 프로젝트 진행 중 프로필 수정 페이지의 기능 구현이 어려웠습니다. 문제는 사용 가능한 API에서 프로필 수정을 위한 HTTP 메서드로 PUT만 제공되었다는 점이었습니다. 일반적으로 PUT 메서드는 기존 데이터를 갱신하는 데 사용되기 때문에, 갱신하기 위해서는 먼저 기존의 데이터를 GET 요청을 통해 가져와야 할 필요성을 느꼈습니다. 이를 해결하기 위해, API 호출 기능을 GET과 PUT으로 분리하여 설정했습니다. 그 후, 각 입력 필드에 맞는 핸들러를 작성하여 문제를 해결할 수 있었습니다.
2. 팔로워 목록 페이지 - useParams를 통한 사용자 ID 추출
  - 팔로워 목록 페이지의 기능 구현 과정에서도 여러 도전에 직면했습니다. 특히, API를 통해 필요한 팔로워 데이터를 호출할 때 연결 문제가 발생했고, 잘못된 데이터를 받아오는 오류도 경험했습니다. 변수 값을 변경해 봤지만, 이는 문제를 해결하지 못했고 때때로는 API 연결 자체가 되지 않는 상황이었습니다. 해결책을 모색하던 중, 팀원들과 함께 설정한 라우터 경로가 사용자 ID를 사용하고 있다는 점에 착안하여, useParams 훅을 사용하여 사용자 ID를 추출하고 이를 기반으로 팔로워 정보를 정확히 가져오는 방법으로 문제를 해결했습니다.
